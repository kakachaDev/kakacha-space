В продукте, над которым я работаю (real-time PWA, много визуальных эффектов), исторически
жил Phaser — целый игровой движок, который использовался ровно для одного: рисовать частицы.
Конфетти, искры, всплески при действиях пользователя.

Это классическая ловушка: «нам нужны частицы → возьмём игровой движок, там они есть».
Работает. Но вместе с эмиттером частиц в бандл приезжает физика, тайлмапы, загрузчик
ассетов, система сцен — сотни килобайт кода, из которых используется процента три.

## Что на самом деле нужно от частиц

Если выписать требования честно, список короткий:

- эмиттер: точка, направление, разброс, частота
- частица: позиция, скорость, размер, прозрачность, время жизни
- рендер: нарисовать N кругов/спрайтов за один кадр
- всё это — стабильно на слабом Android

Ни физики столкновений, ни сцен, ни ассет-пайплайна. Пара сотен строк.

## Две вещи, которые делают разницу

**Пул объектов вместо создания новых.** Частицы рождаются и умирают десятками в секунду.
Если каждый раз создавать объект — сборщик мусора будет регулярно останавливать мир,
и на слабых устройствах это видно как микрофризы ровно в момент «красивого» эффекта.

```js
const pool = []

function spawn(x, y) {
  const p = pool.pop() ?? { x: 0, y: 0, vx: 0, vy: 0, life: 0, size: 0 }
  // ...инициализация полей
  active.push(p)
}

function kill(p) {
  active.splice(active.indexOf(p), 1)
  pool.push(p) // объект вернётся в дело, GC не участвует
}
```

**Один requestAnimationFrame на все эмиттеры.** Не каждый эффект со своим циклом,
а один общий тикер, который обновляет и рисует всё разом. Меньше перерисовок,
предсказуемый порядок слоёв, и паузу для всего сразу сделать тривиально.

## Что получилось

| Было (Phaser) | Стало (свой движок) |
| --- | --- |
| Сотни КБ в бандле ради частиц | Пара сотен строк своего кода |
| Своя сцена и свой canvas движка | Обычный canvas-слой в разметке |
| GC-паузы на слабых устройствах | Пул объектов, аллокаций в кадре нет |
| Обновление движка = риск | Код полностью под контролем |

> Мораль не в том, что игровые движки плохие. Мораль в том, что зависимость надо брать
> под задачу, а не «на вырост». Для частиц в интерфейсе игровой движок — это на вырост.

Кстати, фоновые частицы на этом сайте — тот же подход, только физика считается
в Web Worker. Про это есть [отдельный пост](/blog/2026-07-03-kak-ustroen-etot-sayt).
