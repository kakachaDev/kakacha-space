Когда ты единственный фронтендер на продукте, самый опасный соблазн — «пока и так понятно».
Логика подписок на WebSocket живёт в компоненте, пересчёт баланса — в сторе, обработка
обрыва соединения — в трёх местах сразу. Первые полгода это даже работает.

Потом продукт растёт, и однажды я понял, что не могу ответить на простой вопрос:
«что произойдёт с балансом пользователя, если соединение оборвётся посреди покупки?» —
не прочитав пять файлов. Это был сигнал.

## Решение: отдельный пакет со строгой границей

Я вынес всю бизнес-логику во внутренний SDK — отдельный TypeScript-пакет
(strict mode, без единого `any` на публичной поверхности). Правило границы простое:

> Vue-слой не знает, что такое WebSocket, REST и реконнект.
> SDK не знает, что такое компонент, стор и реактивность Vue.

Внутри SDK:

- **адаптеры транспорта** — REST и WebSocket (SignalR) за общими интерфейсами;
- **менеджер переподключений** — обрыв, backoff, восстановление подписок в одном месте;
- **доменные потоки на RxJS** — балансы и транзакционные события как Observable,
  которые UI просто потребляет;
- **тесты на Vitest** — логика гоняется без DOM и без Vue вообще.

На стороне приложения Pinia-сторы становятся тонкими: подписался на поток из SDK,
положил значение, отдал компонентам.

```ts
// компоненту всё равно, откуда пришло значение —
// из сокета, из REST-снапшота или из кеша после реконнекта
export interface BalanceSource {
  readonly balance$: Observable<Balance>
  refresh(): Promise<void>
}
```

## Что это дало на практике

**Тестируемость.** Сценарий «обрыв посреди транзакции» — это юнит-тест на SDK,
а не ручное «выключи Wi-Fi и молись». Такие тесты ловили реальные гонки.

**Понятность.** Ответ на вопрос «что будет, если…» теперь ищется в одном пакете,
а не по всем компонентам.

**Смену состояния UI-слоя пережили безболезненно.** Проект прошёл через смену подхода
к state-менеджменту — SDK при этом не изменился ни на строчку, потому что он ничего
не знал о том, кто его потребляет. Ради одного этого границу стоило строить.

## Когда так делать не надо

Честности ради: для сайта-визитки или CRUD-админки это оверинжиниринг — слой без
содержания. Граница окупается, когда у домена есть собственная сложность: real-time
события, деньги, состояния, которые живут дольше одного экрана. У нас — есть.
